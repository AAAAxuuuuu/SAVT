# SAVT UI 重构方案文档

## 1. 文档目的

本文档定义 SAVT（Software Architecture Visualization Tool）桌面端 UI 的完整重构方案，覆盖以下内容：

- 产品定位与用户目标
- 当前 UI 问题诊断
- 信息架构与导航模型
- 关键页面线框与交互原则
- 视觉设计系统与 QML-ready token
- 可访问性、性能感知与桌面工作区规则
- 设计决策日志、交付范围与实施优先级

本文档面向产品、设计、Qt/QML 前端实现与架构评审使用，作为后续 QML 重构与视觉落地的统一依据。

## 2. 背景与范围

### 2.1 产品背景

根据 [project-specification-zh.md](./project-specification-zh.md)，SAVT 是一个基于 Qt 6、QML 与现代 C++ 构建的软件架构分析与可视化工具，目标是帮助用户对陌生代码仓库建立系统级认知，而不是成为通用 IDE 或源码阅读器。

### 2.2 本次重构范围

本次重构聚焦桌面端 `apps/desktop` 的 UI/UX，范围包括：

- 应用壳层（app shell）重构
- 左侧导航树与主工作区重构
- L2 能力地图与 L3 组件工作台重构
- 右侧 Inspector 重构
- 顶部全局操作区与 Ask SAVT 入口重构
- 工程分析报告页重构
- 视觉系统与组件层级重构

### 2.3 非范围

本次不以大改 C++ backend 为目标，以下内容默认不在主范围内：

- `AnalysisController` 的业务语义重写
- scene 数据结构的大规模返工
- 布局算法、图分析算法与解析器算法改写
- AI 服务协议变更

允许的后端改动仅限于为了 UI 解耦而增加少量只读属性、命令入口或更清晰的 UI 适配层。

## 3. 当前 UI 诊断

## 3.1 当前实现概况

当前桌面端 QML 主入口为：

- [Main.qml](../apps/desktop/qml/Main.qml)
- [TopControlBar.qml](../apps/desktop/qml/components/TopControlBar.qml)
- [L2GraphView.qml](../apps/desktop/qml/components/L2GraphView.qml)
- [L3ComponentView.qml](../apps/desktop/qml/components/L3ComponentView.qml)
- [RightInspector.qml](../apps/desktop/qml/components/RightInspector.qml)
- [AppTheme.qml](../apps/desktop/qml/components/AppTheme.qml)

其中 [Main.qml](../apps/desktop/qml/Main.qml) 当前约 1281 行，并承担了以下多重职责：

- 应用级布局与页面切换
- L1/L2/L3/L4 视图模型管理
- 选中态与下钻逻辑
- scene 索引与关系辅助计算
- Inspector 数据拼装
- 边绘制与显示过滤
- 局部 tooltip、标题、副标题和行为路由

该文件内存在 60 余个函数，已经超出“页面编排器”的合理职责边界。

## 3.2 主要问题

### 3.2.1 信息密度失衡

当前界面将分析状态、层级切换、图视图、证据链、AI 解释等大量信息同时铺开，尤其右侧 Inspector 过于堆叠，容易让用户在没有稳定主路径的情况下被迫处理过多信息。

### 3.2.2 导航心智不贴近主用户

当前以 `L1-L4` 作为主切换入口，这更接近专业建模视角，而非主用户最关心的：

- 这个项目是做什么的
- 主要有哪些能力块
- 我下一步应该从哪里看起

### 3.2.3 画布与 Inspector 抢焦点

图画布本应是主工作对象，但 Inspector 承担了太多完整内容展示职责，导致中央主视图和右侧面板同时争夺注意力。

### 3.2.4 L2 到 L3 的上下文连续性不足

L2 与 L3 虽然具备功能上的切换，但缺少稳定的 breadcrumb、父级摘要和“为什么进入这个下钻视图”的清晰过渡。

### 3.2.5 视觉层级不够稳

当前卡片、标签、按钮、状态块在视觉上较为接近，导致操作、元数据、容器之间的层级感不强，整体观感更像 demo，而不像专业工程工具。

### 3.2.6 状态反馈不够“工程化”

分析过程中虽然已有进度、阶段和状态信息，但还不够强调“当前正在产出什么”“下一步会发生什么”，会增加等待期间的不确定感。

## 4. 用户与使用场景

## 4.1 用户分层

### 主用户：AI 项目操盘者 / Vibecoder

- 项目能跑，但不了解 AI 生成代码的结构
- 可能不系统掌握工程设计
- 需要快速理解功能、目录、能力和修改切入点

### 次用户：接手改造工程师

- 能读代码，也具备重构能力
- 需要快速建立陌生项目的架构全局认知
- 需要在能力与技术结构之间建立映射

### 第三用户：架构评审者 / 技术负责人

- 关注可追溯、可解释、可展示
- 需要将分析结果带入评审、汇报和跨角色沟通

## 4.2 核心 JTBD

- 当我让 AI 生成了一个项目时，我想快速知道这个项目是做什么的、哪些目录与文件支撑了哪些功能，这样我才能继续安全迭代它。
- 当我拉下一个陌生项目时，我想先看到整体能力地图，再进入重点能力域和组件，这样我才能决定从哪里推进改造。
- 当我发现一个能力域可疑时，我想查看其依赖、证据、关键文件和改造建议，这样我才能判断问题在哪、该不该改。
- 当我准备继续让 AI 帮我修改项目时，我想直接拿到面向当前节点的改造建议和可复制提示词，这样我能把理解转化为行动。
- 当我需要向团队或评审对象讲解项目时，我想生成一份可操作、可追溯的工程分析报告。

## 5. 设计目标与非目标

## 5.1 设计目标

- 降低认知负担，尤其降低首次进入时的信息压迫感
- 降低迷路感，让用户始终知道“我在哪、我在看什么、我接下来能做什么”
- 让 L2 能力地图成为主认知入口，让 L3 成为专业下钻工作台
- 将 Inspector 改造成“摘要优先、详情按需展开”的决策面板
- 提升演示与汇报时的专业感和品牌感
- 提升分析过程中的性能感知与状态反馈可信度
- 保持对专业用户的深度使用能力，包括多选、比较、证据链和报告

## 5.2 非目标

- 不追求一次性覆盖所有高级分析与图交互功能
- 不把 UI 重构做成新的重型业务 backend 项目
- 不用移动端或网页式导航模式替代桌面工作台模式

## 6. 总体设计原则

### 6.1 前门优先（Front-Door Thinking）

主导航应首先回答用户关心的问题，而不是先暴露内部模型层级。对主用户而言，“总览”“能力地图”“报告”比 “L1/L2/L3/L4” 更自然。

### 6.2 渐进披露（Progressive Disclosure）

默认只显示当前任务所需信息。长证据链、完整文件清单、AI 长输出、深层对比信息应通过抽屉、底部托盘或独立工作区承接，而非默认堆在右侧。

### 6.3 识别优于记忆（Recognition over Recall）

用户无需记住当前处于哪个 C4 层级，也不该先理解专业术语才能开始。导航、标题、副标题和摘要文案应帮助识别当前位置。

### 6.4 总览优先，按需下钻（Overview first, zoom and filter, then details-on-demand）

首页与 L2 应优先提供宏观结构，L3 再进入实现层，下钻后通过 Inspector 和底部托盘承接细节。

### 6.5 把理解变成行动

SAVT 的价值不止于“看懂”，而在于让用户快速进入下一步操作。因此 `Ask SAVT`、改造建议和 AI 提示词生成是主路径的一部分，而不是附属工具。

## 7. 信息架构与导航模型

## 7.1 建议信息架构

```text
SAVT
├─ 项目
│  ├─ 总览
│  ├─ 能力地图
│  ├─ 对比
│  └─ 最近分析
├─ 报告
│  ├─ 工程分析报告
│  ├─ 改造建议
│  └─ AI 提示词包
└─ 视图层级
   ├─ L1 系统上下文
   ├─ L2 能力视图
   ├─ L3 组件视图
   └─ L4 全景报告
```

## 7.2 导航结构角色分工

- 左侧导航树：去哪里
- 顶部全局栏：全局操作、项目状态、搜索、Compare、Ask SAVT
- 中央主工作区：当前任务
- 右侧 Inspector：当前对象摘要
- 底部托盘/详情抽屉：长内容、深内容、行动输出

## 7.3 主导航默认落点

- 首次进入项目：`项目 / 总览`
- 主要工作入口：`项目 / 能力地图`
- 深度分析：从能力地图下钻到 `L3 组件工作台`
- 沉淀输出：`报告 / 工程分析报告`
- 专业直达：`视图层级 / L1-L4`

## 8. 关键页面方案

## 8.1 项目总览页

### 页面目标

在 10-20 秒内回答：

- 这个项目是做什么的
- 它有哪些主要能力块
- 建议用户从哪里开始看

### 内容结构

- 顶部：项目名、项目一句话解释、重新分析、Ask SAVT
- 左上：项目摘要卡
- 中部：主要能力块卡组（4-6 个）
- 左下：目录到能力映射概览
- 右下：推荐阅读路径与快速操作

### 关键原则

- 不默认展示大图画布
- 不要求用户先理解层级术语
- 直接给出“下一步看哪里”的行动建议

## 8.2 L2 能力地图

### 页面目标

帮助用户理解这个项目由哪些能力构成，以及这些能力如何关联。

### 默认视角

L2 默认采用“业务能力地图”，而不是“技术模块图”。技术视角可作为切换模式保留。

### 页面结构

- 顶部：breadcrumb、视角切换、过滤、搜索、fit、focus、multi-select
- 中央：能力图主画布
- 右侧：摘要型 Inspector
- 底部：选中上下文条与下一步动作

### 关键原则

- 图是主角
- Inspector 只做摘要
- 选中后的下一步动作要明显

## 8.3 L3 组件工作台

### 页面目标

帮助用户从能力域进入组件级理解，并形成具体改造切口。

### 页面结构

- 顶部：breadcrumb、返回能力地图、过滤器、路径追踪、Compare、Ask SAVT
- 中央：组件关系图
- 右侧：当前组件摘要
- 底部：相关文件、依赖路径、AI 提示词草案、对比结果

### 关键原则

- 明确保留父级能力上下文
- 强调“依赖理解 -> 改造建议 -> AI 提示词”这条动作链

## 8.4 右侧 Inspector

### 默认摘要区块

- 这是什么
- 为什么重要
- 关键文件
- 关系解释
- 下一步动作

### 详情迁移策略

以下内容不再默认塞入右栏：

- 长文件列表
- 完整证据链全文
- AI 长输出
- 大段说明文字
- 深层比较内容

这些内容应迁移到详情抽屉或底部托盘。

## 8.5 工程分析报告页

### 页面目标

为工程师提供可操作、可追溯、可继续执行的分析结果。

### 报告应包含

- 项目摘要与分析上下文
- 关键能力与目录/文件映射
- 主要依赖与风险
- 证据链与结论
- 改造建议与 AI 提示词入口

### 关键原则

报告不是静态海报，而是可继续跳回分析流的操作页面。

## 9. 交互设计原则

## 9.1 L2 / L3 画布

- 单击节点：选中并更新 Inspector 摘要
- 双击节点：下钻或打开深层上下文
- `Shift + 点击/框选`：多选比较
- 滚轮：以光标为中心缩放
- `Space + 拖动` 或中键拖动：平移
- `Esc`：退出聚焦或清理临时状态

## 9.2 Ask SAVT

Ask SAVT 为主路径动作，应支持：

- 解释当前节点
- 生成改造建议
- 生成可复制给 AI 的提示词

入口位置：

- 顶部全局入口
- 节点悬停快捷动作
- Inspector 内下一步动作区

## 9.3 Compare

优先支持两类比较：

- 两次分析结果比较
- 两个文件的功能归属解释比较

Compare 不建议一开始就深嵌在右栏，而应拥有独立视图或上下文条。

## 10. 视觉设计系统

## 10.1 视觉方向

### 气质定位

- 对主用户：更像架构驾驶舱

## 11. 桌面工作区与布局规则

### 11.1 建议尺寸

- 顶栏高度：56-64 px
- 左侧导航宽：232-248 px
- 右侧 Inspector 宽：344-380 px
- Inspector 折叠态：64-72 px
- 底部托盘高度：180-240 px
- 主内容最小宽度：960 px

### 11.2 适配规则

- 1920+：可并行显示主视图、Inspector 和底部托盘
- 1536-1919：保持 Inspector，底部托盘按需展开
- 1280-1535：Inspector 可默认折叠
- 小于 1280：不作为主设计分辨率目标

## 12. 可访问性与包容性设计

- 文字与背景遵循 WCAG 2.1 AA 对比度
- 节点类型不能仅靠颜色区分，需叠加形状、图标或边框样式
- 必须提供显著 focus ring
- 键盘路径必须贯通导航、主工作区、Inspector 和详情区
- 错误、阻塞、语义不可用状态必须可读，不仅靠色块表达

## 13. 性能感知设计

性能感知与真实性能同样重要，建议：

- 分析阶段明确显示当前阶段与产物
- L2/L3 在分析中使用骨架态或空态说明，而非闪空白
- 大图中默认使用聚焦模式与筛选，而不是一开始渲染所有高复杂关系
- 搜索命中后轻量定位，不强制跳屏

## 14. 设计决策日志

### 决策 1：主导航不再以 L1-L4 作为前门

- 备选：继续沿用 L1-L4 主切换
- 采用：`项目 / 报告 / 视图层级`
- 原因：更符合主用户心智，降低认知门槛
- 代价：专业用户需要额外点击进入层级视图
- 缓解：保留 `视图层级` 作为稳定专业入口

### 决策 2：L2 默认使用业务能力地图

- 备选：技术模块图作为默认
- 采用：业务能力地图默认，技术视角作为切换
- 原因：主用户首先关心“项目做什么”
- 代价：部分专业用户希望直接看技术图
- 缓解：提供视角切换与专业入口

### 决策 3：Inspector 采用摘要优先

- 备选：维持完整详情堆叠
- 采用：摘要优先 + 详情迁出
- 原因：解决当前最显著的认知负担问题
- 代价：深内容查看需多一步操作
- 缓解：提供明确“展开详情”与底部托盘

### 决策 4：L3 定义为上下文工作台

- 备选：继续作为普通页面切换项
- 采用：从 L2 下钻进入、保留父能力上下文
- 原因：增强任务连续性
- 代价：结构实现比简单页面切换更复杂
- 缓解：通过 app shell 与 workspace 容器管理状态

## 15. 实施优先级

### P0：应用壳与主路径

- 左侧导航树
- 顶部全局栏
- 项目总览页
- L2 能力地图主页面
- Inspector 摘要态

### P1：深度工作流

- L3 组件工作台
- Ask SAVT 多入口整合
- 详情抽屉 / 底部托盘

### P2：沉淀输出与增强

- 工程分析报告页
- 改造建议页
- AI 提示词包页
- 深度比较模式

## 16. 风险与开放问题

- Compare 最终应更偏“版本对比”还是“归属解释对比”，需在实现前进一步收口
