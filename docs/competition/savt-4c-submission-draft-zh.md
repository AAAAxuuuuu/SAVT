## 2. 摘要

随着软件系统规模持续扩大，开发者面对的不再只是单个文件或单个函数，而是跨目录、跨模块、跨语言、跨构建环境的复杂工程。传统文件树、全文检索和普通依赖图能够提供局部线索，却难以回答“系统从哪里进入”“核心能力如何分布”“模块之间为什么存在依赖”“当前分析结论是否达到语义级精度”等架构理解问题。SAVT 面向这一痛点，设计并实现了一套源码事实驱动的软件架构分析与可视化系统。

SAVT 以 Qt 6、QML 和 C++20 为基础，构建了从项目扫描、语法分析、语义诊断、图模型聚合、分层布局到桌面交互展示的完整链路。系统当前能够识别 C/C++、QML、JavaScript、Python、Java、JSON、CMake 等架构相关文件，基于 Tree-sitter 抽取 C/C++ 符号、包含关系和初步调用关系，并在启用 LLVM/Clang 条件时进入 libclang 语义路径。系统明确提供 `SyntaxOnly`、`SemanticPreferred` 和 `SemanticRequired` 三种精度模式，当语义条件不满足时，会输出 `missing_compile_commands`、`backend_unavailable`、`llvm_not_found`、`system_headers_unresolved` 等具体诊断，避免用启发式结果冒充工业级结论。

在展示层，SAVT 以 L1 系统上下文、L2 能力域图、L3 组件下钻和 L4 Markdown 报告组织架构认知，并为节点和关系边附带事实、规则、结论、来源文件、符号、模块和置信原因等证据包。系统还实现了扫描、解析、聚合、布局四层增量缓存，使重复分析同一工程时能够复用稳定结果。测试体系覆盖 21 个精度样例和 21 组黄金结果，包含后端、快照、AI、UI 服务层与性能基线测试。实验记录显示，在 10001 文件合成工程中，热缓存分析耗时由冷启动约 4519.41ms 降至约 303.01ms。

本作品的创新点在于把“能画图”推进到“能解释、能追溯、能声明精度边界”。SAVT 不是通用聊天助手，也不是静态图片生成器，而是面向软件工程实践的架构理解工具。它让开发者在接手项目、准备评审、规划重构和撰写技术文档时，能够从源码事实出发快速形成可信的系统认知。

关键词：软件架构可视化；静态分析；语义分析；C++；Qt；Tree-sitter；libclang；证据链；增量缓存

## 4. 选题背景与问题定义

现代软件项目的复杂度正在从“代码量大”转向“结构难懂”。一个中大型项目往往包含多个入口、多个层次、多个框架、多个构建目标和长期演进留下的历史目录。新成员接手项目时，通常先在文件树、README、搜索结果和零散图表之间来回切换。即便项目有文档，也常常存在滞后、缺失或与真实代码不一致的问题。对于学生团队、课程项目、开源协作和企业研发来说，理解系统结构所消耗的时间已经成为开发效率和质量的重要限制。

传统工具能解决部分问题。IDE 可以跳转定义，全文检索可以定位关键字，依赖分析工具可以列出包含关系，通用大模型可以解释一段代码。然而，这些工具往往缺少统一的系统视角。它们无法稳定回答“哪些模块是入口”“哪些能力域承担核心职责”“一个节点的结论来自哪些文件和符号”“当语义条件不满足时结论还能不能信”等问题。对 C/C++ 工程而言，如果没有编译数据库和语义前端，仅凭文件文本很容易误判重载、模板、宏展开、跨翻译单元符号和真实调用关系。

SAVT 将问题定义为：如何从源码事实出发，为一个项目生成可解释、可追溯、可交互的软件架构认知视图。系统既要降低初学者和新成员的理解门槛，又要对精度边界保持诚实；既要生成可看的图，也要生成可审查的证据；既要能展示全局，又要能支持局部下钻。

## 5. 需求分析

SAVT 的核心用户是需要快速理解代码仓库结构的人。学生团队在比赛、课程设计和科研项目中需要向评委或导师解释系统结构；企业新成员需要在有限时间内了解项目入口、核心模块和依赖方向；架构师和重构负责人需要判断模块边界、依赖风险和演进路线；开源维护者需要把复杂项目转化为新贡献者可以读懂的地图。

围绕这些用户，系统需要满足五类需求。第一是项目扫描需求，能够递归识别架构相关文件，并过滤构建产物、第三方依赖和工具缓存。第二是源码分析需求，至少能从 C/C++ 代码中抽取符号、包含关系、继承关系、调用关系和类型使用关系，并能在具备编译数据库时进入语义分析。第三是架构聚合需求，能够把文件和符号归并成模块、能力域、组件和依赖链路。第四是交互展示需求，用户需要从系统上下文进入能力图，再下钻到组件图，并查看报告、AST 和证据链。第五是可信解释需求，系统必须说明每个结论的事实来源、规则依据、置信原因和精度状态。

非功能需求同样重要。系统需要跨平台构建，适配 Windows 与 macOS；需要具备自动化测试和性能基线；需要保护本地密钥和项目隐私；需要在 AI 能力不可用时保持核心分析功能可用；需要把第三方依赖和自研代码边界分清楚。

## 6. 总体设计

SAVT 采用分层架构设计，核心原则是分析事实与展示界面解耦。底层由 `src/parser` 与 `src/analyzer` 负责源码解析和关系抽取；中间由 `src/core` 负责统一图模型、能力图、组件图、项目配置和证据包；上层由 `src/layout` 负责图布局；再往上由 `src/ui` 负责 Qt 控制器、增量分析管线、场景映射、报告服务、AST 预览和 AI 服务；最外层由 `apps/desktop` 提供 QML 桌面工作台。

```text
Desktop QML Workbench
  -> Qt UI Controller and Services
  -> Incremental Analysis Pipeline
  -> Syntax or Semantic Analyzer
  -> Core Architecture Models
  -> Layout Engine
  -> Scene Mapping
  -> Interactive Views, Reports, AST Preview, AI Explanation
```

一次完整分析从用户选择项目目录开始。系统首先构建扫描清单和元数据指纹，再根据精度模式决定是否探测 `compile_commands.json` 和语义后端。如果条件满足，系统进入 libclang 语义路径；如果条件不满足，系统根据用户选择决定回退到 Tree-sitter 语法路径或生成阻断报告。分析结果进入 `AnalysisReport` 后，系统继续生成 `ArchitectureOverview`、`CapabilityGraph`、`ComponentGraph` 和规则报告，并通过 `LayeredGraphLayout` 计算可视化布局。最后，`SceneMapper` 将 C++ 数据模型映射为 QML 可直接渲染的数据结构。

这一设计让 SAVT 的每个层次都承担明确职责。UI 只消费数据，不在渲染时推断业务事实；布局只计算位置，不改变分析结论；AI 只解释证据包，不扫描整个仓库；核心模型保留精度状态，使所有展示和报告都能回到同一事实底座。

## 7. 关键技术实现

项目扫描模块以根目录为输入，识别 C/C++、QML、JavaScript、Python、Java、HTML、CSS、JSON、CMake 等架构相关文件。扫描器默认跳过 `build`、`third_party`、`node_modules`、Qt 自动生成目录、缓存目录和锁文件，并支持通过 `config/savt.project.json` 声明忽略目录、模块归并、角色覆盖、入口覆盖和依赖折叠规则。这种设计把项目特例从代码中抽离出来，使分析规则可以根据被分析项目调整。

语法分析模块基于 Tree-sitter C++。它将 C/C++ 源文件解析为语法树，抽取命名空间、类、结构体、函数、方法、文件级包含关系和初步调用关系。Tree-sitter 的优势在于速度快、容错性好，适合在缺少编译环境时提供架构级初始视图。SAVT 还在桌面端提供 AST 预览，让用户能查看某个 C++ 文件的语法结构，辅助理解分析结果。

语义分析模块基于 libclang。与单纯文本分析不同，语义分析依赖目标工程的真实编译命令。SAVT 在 `SemanticPreferred` 模式下会优先尝试语义后端，失败后回退到语法路径；在 `SemanticRequired` 模式下，如果没有 `compile_commands.json`、LLVM 不可用、头文件缺失或翻译单元解析失败，系统会直接阻断并给出诊断。这样的设计把“分析能力”与“精度声明”绑定起来，避免误导用户。

图模型聚合模块负责把底层符号转化为架构概念。`AnalysisReport` 保存符号节点和关系边，`ArchitectureOverview` 形成模块视角，`CapabilityGraph` 形成能力域视角，`ComponentGraph` 支持单个能力域内部下钻。每个能力节点都可以记录职责、摘要、示例文件、协作者、技术标签、风险信号和证据包。证据包包含事实、规则、结论、来源文件、符号、模块、关系、置信标签和置信原因，是系统可解释性的基础。

布局模块通过分层图布局把抽象关系转化为屏幕上的空间组织。L2 能力图帮助用户把握系统主干，L3 组件图帮助用户查看某个能力域内部结构。桌面端 `ArchitectureCanvas.qml` 支持缩放、平移、全景视图、节点聚焦、入边和出边关系聚焦，以及从能力节点下钻到组件探测界面。用户可以先看全局，再沿着关系边逐步阅读。

增量缓存模块将一次分析拆为扫描、解析、聚合和布局四层。缓存键绑定项目路径、分析选项、项目配置、编译数据库、源码指纹和缓存版本。第二次分析同一项目时，如果源码、配置和精度模式没有变化，系统可以复用已有结果，显著减少等待时间。该模块已经有 UI 服务层回归测试保护，包括稳定层缓存命中和源文件变更后的失效行为。

AI 辅助解释模块采用受约束设计。系统不会把整个仓库直接交给模型，而是构建有限的架构证据包，发送给 DeepSeek 官方 API 或 OpenAI 兼容代理端点。内置提示词要求模型只处理 SAVT 架构阅读任务，只使用提供的证据，不臆造文件、模块或业务意义，并以结构化 JSON 返回。AI 输出服务于辅助阅读，不替代静态分析事实。

## 8. 系统功能

SAVT 桌面端提供项目选择、分析启动、停止分析、语义状态展示、架构全景图、组件探测、报告查看、AST 预览和 AI 解释入口。用户可以从命令行传入项目路径，也可以在界面中选择文件夹。分析过程会显示阶段和进度，完成后生成系统上下文卡片、能力图、组件图和 Markdown 报告。

L1 系统上下文用于回答“这个项目是什么”。它总结项目名称、入口、主要容器、输入输出、技术栈、精度状态和阅读顺序。L2 能力图用于回答“系统由哪些能力构成”。它把模块聚合为入口、核心能力和基础设施节点，并展示能力之间的激活、支撑和基础设施使用关系。L3 组件下钻用于回答“某个能力内部怎么协作”。它以组件节点和协调关系呈现局部结构。L4 报告用于回答“如何把分析结果带出系统”。它输出适合复制、归档和进一步评审的 Markdown 文本。

系统还提供语义分析状态页。当用户选择语义模式但环境不满足时，界面不会静默失败，而是说明缺少编译数据库、后端不可用、LLVM 未找到、头文件无法解析或翻译单元解析失败等具体原因。对于 C/C++ 架构分析，这种诊断能力非常关键，因为真实工程中语义条件经常比算法本身更难满足。

## 9. 测试与实验结果

SAVT 当前包含后端测试、快照测试、AI 测试、UI 服务层测试、性能基线测试和 UTF-8 编码检查。测试目标分别为 `savt_backend_tests`、`savt_snapshot_tests`、`savt_ai_tests`、`savt_ui_tests`、`savt_perf_smoke` 和 `savt_utf8_check`。快照测试维护 21 个精度夹具和 21 组黄金结果，覆盖 C++ 分层项目、C++ 插件宿主、Header-only 库、vendored dependency、Node 后端、Python 工具项目、QML 混合项目、Spring Boot 项目，以及语义成功与语义阻断场景。

性能基线记录显示，代表性小项目分析耗时在毫秒级。`cpp_vendored_dependency_app` 样例发现并解析 6 个文件，冷启动总耗时约 3.08ms，热缓存约 0.45ms；`qml_dashboard_workspace` 样例发现 8 个文件、解析 5 个文件，冷启动约 3.00ms，热缓存约 0.54ms。对于 10001 文件的合成工程，冷启动总耗时约 4519.41ms，热缓存总耗时约 303.01ms，四层缓存均命中，峰值内存约 75.59MB。这说明系统在重复分析、迭代调试和演示场景中具备较好的响应潜力。

语义精度测试方面，项目已经覆盖缺失编译数据库、后端不可用、LLVM 未找到、系统头文件无法解析和跨翻译单元语义合并等路径。阶段记录显示，启用语义能力的环境中，`cpp_semantic_cross_tu` 可验证模板类型跨翻译单元 identity 归并；在语义条件不足时，系统也能输出清晰阻断状态。当前未在本文档生成时重新执行完整构建测试，因此最终参赛提交前仍建议在参赛演示机器上重新跑通标准测试脚本。

## 10. 创新点

第一，SAVT 提出了精度声明与运行条件绑定的分析机制。很多架构可视化工具在缺少编译上下文时仍然输出看似完整的图，但这些图可能只是文本启发式结果。SAVT 把 `SyntaxOnly`、`SemanticPreferred` 和 `SemanticRequired` 写入模型和报告，当语义条件不满足时给出具体状态码。这种机制让用户知道自己看到的是语法级推断、语义级事实，还是被环境阻断的结果。

第二，SAVT 将架构图与证据包结合。系统不仅展示节点和关系，还为节点保存事实、规则、结论、来源文件、符号、模块、关系和置信原因。评审人员可以追问“为什么这个模块被归为入口”“为什么这条边存在”“这个结论来源于哪些文件”，系统都有统一的证据结构可供展开。这让架构图从展示材料转变为可审查对象。

第三，SAVT 采用 L1-L4 的渐进式理解路径。用户先通过系统上下文建立整体认知，再进入能力域图查看主干结构，然后下钻到组件图分析局部协作，最后通过 Markdown 报告沉淀结果。这种路径符合人类阅读复杂系统的认知过程，也适合比赛答辩从宏观到微观展开。

第四，SAVT 实现了面向架构分析的四层增量缓存。扫描、解析、聚合和布局分别缓存，源码、配置或编译数据库变化时按层失效。相比每次从零开始，分层缓存能显著改善二次分析体验，也为未来的实时架构观察和版本对比奠定基础。

第五，SAVT 对 AI 辅助解释设置了边界。系统只发送有限证据包，要求模型只做架构阅读任务，并以结构化结果返回。这避免了把 AI 当作无边界聊天助手，也降低了虚构文件、泄露密钥和偏离分析事实的风险。

## 11. 应用价值

在教学场景中，SAVT 可以帮助软件工程课程把“架构”从抽象概念落到真实代码。学生不仅能看到模块图，还能追溯图背后的文件和符号，从而理解分层、依赖、入口和组件协作。对于课程设计和毕业设计，SAVT 还能辅助生成项目说明、系统上下文和架构报告。

在团队研发场景中，SAVT 可以缩短新成员接手项目的时间。新成员通常需要先理解系统入口、核心模块和依赖方向，而 SAVT 能把这些信息组织成可交互地图。对于重构评审，SAVT 可以帮助团队识别高风险依赖、历史目录、第三方边界和语义条件不足的问题。

在开源协作场景中，SAVT 可以为维护者提供自动化的架构导览材料。开源项目文档常常滞后，SAVT 以源码为输入生成视图和报告，可以帮助贡献者更快找到阅读起点。对于大型 C/C++ 项目，语义模式和编译数据库诊断尤其有价值。

## 12. 可行性与工程成熟度

SAVT 的可行性来自已经落地的工程实现。仓库当前拥有明确的模块划分、可运行的桌面入口、构建预设、项目级配置、Tree-sitter 解析路径、可选 libclang 语义路径、图聚合模型、布局引擎、QML 工作台、AI 适配器、快照测试和性能基线。自研扫描范围中 C/C++ 文件约 19184 行，QML 与 JavaScript 文件约 12200 行，测试体系覆盖 21 个命名精度样例。

项目也保留了真实的工程边界。`third_party/` 被视为外部依赖，不纳入自研代码；`config/*.local.json` 被 `.gitignore` 排除，用于保护 API 密钥；`CMakeUserPresets.json` 和 Qt/CMake 生成物被排除，避免本地环境污染仓库；Windows、macOS 和 MinGW 兼容路径通过 CMake Presets 管理。

## 13. 知识产权、数据安全与 AI 使用说明

SAVT 自研代码主要位于 `src/` 与 `apps/`，第三方依赖主要为 Tree-sitter、tree-sitter-cpp、Qt 和可选 LLVM/Clang 相关库，参赛提交时应按各自许可证要求附带开源声明。项目不会把本地密钥文件提交到仓库，AI 配置应使用 `config/deepseek-ai.local.json` 保存，并由 `.gitignore` 排除。

SAVT 的 AI 功能是可选辅助功能，不是核心静态分析路径。系统只在用户手动触发时调用外部 API，并发送受限的证据包、模块摘要和用户任务，不默认上传整个仓库。若比赛要求声明 AI 工具使用情况，应说明作品运行时支持 DeepSeek 官方 API 或 OpenAI 兼容代理端点，AI 输出仅用于解释静态分析结果，最终事实依据仍来自源码扫描、语法/语义分析和证据包。

如果比赛现场无法联网，SAVT 仍可演示项目扫描、架构图、组件下钻、AST 预览、语义状态和报告生成。AI 解释卡片可作为可选能力展示，不影响核心功能闭环。

## 14. 当前不足与后续计划

SAVT 当前最大的不足是语义精度仍受环境约束。真正的工业级 C++ 分析需要目标项目提供完整 `compile_commands.json`，并确保 LLVM/Clang、系统头文件和编译参数可解析。后续计划是在 Windows 和 macOS 上分别准备稳定的语义演示环境，扩展模板、重载、宏、别名、继承和跨翻译单元样例。

第二个不足是 QML 资源治理仍在迁移中。当前主工作台使用 `greenfield` QML 文件，但仓库中仍保留一批旧组件、旧页面和旧工作区文件。后续应明确哪些文件继续维护、哪些文件转入归档、哪些文件清理，以降低参赛打包和长期维护成本。

第三个不足是多语言语义深度不均衡。SAVT 已能识别多语言工程结构，但语义级深度主要集中在 C/C++。后续可以针对 JavaScript/TypeScript、Python 和 Java 分别引入更成熟的语言服务或解析器，提升跨技术栈项目的分析可信度。

第四个不足是演示材料仍需补齐。参赛前应准备一个稳定的演示项目、离线演示视频、10 分钟答辩脚本、核心流程截图、安装包、开源许可证清单和测试运行截图。这样可以让评审材料与现场演示互相印证。

## 15. 结论

SAVT 面向复杂代码仓库的架构理解问题，构建了一套从源码事实到交互式架构认知的完整系统。它通过 Tree-sitter 语法分析、可选 libclang 语义分析、统一图模型、证据包、分层布局、增量缓存和受约束 AI 解释，把“读代码”转化为“读架构”。项目已经具备可运行桌面端、清晰工程分层、自动化测试、性能基线和较完整文档体系。

对于中国大学生计算机设计大赛的软件应用与开发类作品而言，SAVT 的价值不在于展示一个炫目的关系图，而在于提供可信、可追溯、可解释的软件工程工具。它能够帮助学生、开发者和架构评审人员更快理解系统结构，也能把复杂工程中的隐性知识转化为可共享的架构材料。后续随着语义样例、跨语言能力和打包交付继续完善，SAVT 有潜力成为面向软件工程教育与实践的高价值架构分析平台。
