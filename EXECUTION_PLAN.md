# SAVT 修订执行计划

更新日期：2026-04-26

## 产品目标

SAVT 的目标不是做一个 IDE，也不是做一个单文件阅读器。

它要解决的问题是：

- 面对越来越大的项目，尤其是 AI 辅助生成、迭代很快、局部可读但整体失焦的代码库，帮助用户快速建立“项目整体结构”的心智模型。
- 先把项目拆成可理解的架构层级，再辅助用户下钻阅读代码，而不是要求用户从文件树开始一层层手工读。
- 以 C++ 为第一优先级，后续逐步扩展到 Java 和其他面向对象语言。

当前阶段的判断标准也应该围绕这个目标，而不是围绕“界面是否好看”。

## 规划原则

这份修订计划遵循四条原则：

1. 先做可信度，再做美观度。
2. 先做证据链，再做更多 AI 包装。
3. 先把构建、精度、数据模型收口，再做缓存。
4. 重构阶段只做等价重构，不顺手改行为。

## 补充产品方向：分层分析模式

针对“大项目一次全量分析太慢、用户只想先看核心框架”的实际使用场景，后续需要补一条明确的产品能力线：

- 支持“逐步分析”，先给出核心框架和主干模块的初始结果。
- 支持“局部分析”，允许用户按目录、模块、能力域或入口点继续下钻。
- 支持“全量分析”，在用户明确需要完整结果时再执行整仓扫描与语义增强。

这条能力线当前先记录为产品规划，不作为本轮立即插队实现项。

后续落地时建议拆成三层：

- 第一层：快速模式，只分析核心目录、入口链路和主模块骨架，优先保证首屏反馈速度。
- 第二层：局部模式，基于用户选中的模块或目录追加分析，而不是重新全量开始。
- 第三层：完整模式，执行全量语义分析、完整关系聚合和最终布局。

验收方向：

- 用户第一次分析大项目时，不必等待全量完成才能看到可用结构。
- 分析模式需要在 UI 上显式可见，而不是隐式切换。
- 这条能力应与后续项目级配置、缓存和增量分析协同设计，而不是单独堆功能。

## 后续想法：后台高精度预热

UI 调整完成后，评估并实现一条“高精度后台预热”能力线：

- 普通快速分析启动或完成后，后台自动以低占用方式准备一轮 `SemanticPreferred` 高精度分析。
- 预热任务默认不抢 UI 状态，不频繁刷新前端，只把结果写入缓存或暂存区。
- 预热期间限制资源占用：单任务、低优先级、可取消、分阶段检查、避免重复生成 `compile_commands.json`。
- 用户点击“高精度”时，如果预热已经完成，直接应用预热结果；如果仍在运行，则接管为前台任务并允许全速完成；如果项目或配置已变化，则取消旧任务重新开始。
- 该能力需要单独管理后台 watcher/状态，避免和当前前台 `QFutureWatcher` 分析链路互相覆盖。

建议分两步落地：

1. 先做后台高精度预热和缓存，不自动切换 UI。
2. 再做点击高精度时复用、接管或提速后台任务。

## 当前状态判断

基于当前仓库现状，项目大致处于以下位置：

- 第 0 阶段已完成当前验收：clean clone 可配置，macOS 基线可复现，Windows 入口已标准化。
- 第 1 阶段已完成当前验收：语义后端探测、阻断、报错链路已标准化。
- 第 2 阶段已完成当前验收：semantic-capable macOS 环境已完成实机验证，跨 TU 模板语义归并与相关快照基线已收口。
- 第 3 阶段已完成当前验收：仓内 21 组 precision 样本、默认基线 17 组快照、15 组审计断言已建立。
- 第 4 阶段已完成当前验收：`AnalysisController` 已从 2221 行收口到 825 行，旧的分析编排/报告/AI/AST 重复实现已移除，结果组装统一回到 `AnalysisOrchestrator`。
- 第 5 阶段已完成当前验收：capability scene 已统一为单一 DTO，layout 成为 L2 坐标和边路由的唯一事实来源。
- 第 6 阶段已完成当前验收：capability node/edge evidence package 已由 core 层生成并透传到 scene，inspector 已按“事实 -> 规则 -> 结论”展示，并支持关系级证据下钻。
- 第 7 阶段已完成当前验收：L3 已替换 AST 占位页，能够从 L2 能力域下钻到内部组件图，并支持返回、过滤、聚焦和组件级证据展示。
- 第 8 阶段已完成当前验收：项目级配置系统已支持忽略目录、模块归并、角色覆盖、入口覆盖和依赖折叠，SAVT 仓库自身也已落地默认项目配置。
- 第 9 阶段已完成当前验收：自动化测试已扩展到 UI 服务层，并新增了可重复执行的性能基线工具和 10k+ 文件 full baseline 记录。
- 第 10 阶段已完成当前验收：扫描、解析、聚合、layout 四层同进程增量缓存已落地，cache key 已显式绑定文件指纹、编译数据库、项目配置和 analyzer 版本，并完成 cold/hot baseline 验证。

因此，当前项目不应被视为“还停留在第 7 阶段”。

更准确地说，它处于“前 10 个阶段已完成，第 11 阶段待启动”的状态。

## 阶段重排说明

主要调整如下：

- 把“结果可追溯能力”前置到真实 L3 之前。
- 把“增量分析和缓存”后置到数据模型和规则体系稳定之后。
- 补强第 0 阶段，明确要求 clean clone 可构建，而不仅仅是本地能跑。

## 修订后的执行计划

### 第 0 阶段，1 到 3 天：工程卫生与可复现构建收口

目标是把“这台机器能跑”提升为“干净仓库也能复现”。

动作：

- 清理生成物、IDE 噪音文件、误跟踪的本地配置。
- 修正 `.gitignore` 和仓库跟踪策略，移除不该进仓库的生成文件。
- 明确 `third_party/` 的来源策略，避免 clean clone 后直接缺依赖。
- 固定最小可重复的 configure/build/test 入口，至少覆盖主开发平台。
- 记录当前 baseline，包括成功项和已知失败项，而不是只记录“理想状态”。

验收标准：

- 仓库主线没有生成物和本地 IDE 噪音。
- clean clone 后按文档步骤可以完成 configure。
- 至少有一条统一的 build/test 入口能被持续复用。

### 第 1 阶段，3 到 5 天：语义后端入口、探测和阻断做扎实

目标是让语义模式的行为可信，而不是“看起来像支持”。

动作：

- 完整打通 `SAVT_ENABLE_CLANG_TOOLING` 的 CMake 探测链路。
- 明确区分以下失败类型：
  - 未安装或未找到 LLVM/Clang
  - 找不到 `compile_commands.json`
  - 编译数据库为空
  - 系统头无法解析
  - translation unit 解析失败
- 保证 UI、日志、报告中的语义状态一致。
- `SemanticRequired` 模式下禁止模糊降级。

验收标准：

- 语义模式要么成功进入 backend，要么明确阻断并解释原因。
- 失败状态可以稳定复现，不依赖人工猜测。

### 第 2 阶段，5 到 8 天：把语义抽取能力做成可验证的后端

目标是把 `SemanticProjectAnalyzer` 从“有骨架”推进到“可依赖的核心能力”。

动作：

- 继续完善符号注册、跨 TU 合并、边收集、去重归并。
- 以 `USR` 和语义声明为主 identity，不靠名字拼接作为主路径。
- 稳定处理 declarations、definitions、overrides、inherits、uses-type、calls。
- 明确 `FactSource::Semantic` 和 `FactSource::Inferred` 的边界。
- 补 semantic-capable 样本，覆盖成功、阻断和 fallback。

验收标准：

- 跨 TU 符号和边关系能稳定复现。
- 报告里能明确区分推断事实和语义事实。
- 语义路径的输出不再只是“尝试过”，而是“有可用结论”。

### 第 3 阶段，4 到 7 天：建立精度基准与回归基线

目标是以后改算法时，能知道自己是在进步还是退步。

动作：

- 扩展现有 `tests/fixtures/precision/` 和 `tests/golden/precision/`。
- 从现有样本起步，逐步增加到 20 个以上代表性项目。
- 对类、函数、调用、继承、类型依赖、模块边建立人工核对样本。
- 对 `AnalysisReport` JSON、Overview、CapabilityGraph、Markdown、DOT 保持 golden 测试。
- 区分 syntax path 和 semantic path 的期望输出。

验收标准：

- analyzer、overview、capability、layout 改动都能量化比较。
- 测试输出能回答“稳定性是否变化”和“精度是否变化”。

### 第 4 阶段，3 到 5 天：完成 `AnalysisController` 等价重构

目标是让 UI 控制器回到“状态和动作入口”的角色。

动作：

- 继续下沉 `AnalysisController` 里的分析编排、报告拼装、AST 预览、AI 适配、scene 转换。
- 巩固以下服务边界：
  - `AnalysisOrchestrator`
  - `ReportService`
  - `AstPreviewService`
  - `AiService`
  - `SceneMapper`
- 不在这个阶段顺手改 UI 行为和产品语义。

验收标准：

- `AnalysisController` 不再承担核心分析逻辑。
- 现有 UI 行为不变。
- 重构后的模块边界更容易测试和继续演进。

### 第 5 阶段，2 到 4 天：统一布局和 scene 事实源

目标是让布局坐标、分组和边路由只从一个地方产生。

动作：

- 抽出统一 scene DTO。
- 让 L2 及后续 L3 都从 layout/service 产物获取坐标、分组、边路线。
- 清理 UI 层和 controller 层中残留的“手工拼坐标”逻辑。
- 明确哪些数据属于 graph，哪些属于 scene，哪些属于 presentation。

验收标准：

- UI 不再自己制造坐标事实。
- layout 层成为唯一坐标来源。
- scene 组装逻辑可复用到 L2 和未来 L3。

### 第 6 阶段，4 到 6 天：补齐结果可追溯能力

目标是让工具的结论可解释，而不只是“看起来合理”。

动作：

- 给 node、edge、capability 增加 evidence package。
- evidence 至少包含：
  - 来源文件
  - 相关符号
  - 上游边或参与关系
  - 命中的规则
  - 可信度或置信提示
- 右侧 inspector 按“事实 -> 规则 -> 结论”展示。
- AI 解读继续作为附加层，不替代证据链。

验收标准：

- 任何一个图元素都能回答“为什么这么判断”。
- 用户可以从图回溯到源文件和符号证据。

### 第 7 阶段，4 到 7 天：把 L3 真正做出来

目标是从“全局图”走向“单能力域内部结构图”。

动作：

- 定义从 capability 下钻到 component 的数据模型。
- 先支持单个 capability 的内部组件图。
- 增加 breadcrumb、返回、过滤、聚焦。
- 节点和边都可连接到第 6 阶段的 evidence package。

验收标准：

- 用户从 L2 点击一个能力域后，可以看到它的内部组件关系。
- L3 不再是 AST 预览或占位内容。

### 第 8 阶段，3 到 5 天：引入项目级配置系统，削弱硬编码启发式

目标是让项目特例通过配置纠正，而不是每次改源码规则。

动作：

- 在项目根或 `config/` 下引入项目配置文件。
- 支持：
  - 忽略目录
  - 模块归并
  - 角色分类
  - 入口识别
  - 依赖折叠规则
- 让 `ArchitectureOverview` 和 `CapabilityGraph` 从“字符串规则优先”逐步过渡为：
  - 语义事实
  - 配置规则
  - 项目特征模型

验收标准：

- 用户能在不改源码的情况下纠正项目特例。
- 字符串启发式不再是唯一事实来源。

### 第 9 阶段，4 到 6 天：补齐测试矩阵和性能基线

目标是给每个核心模块建立自动化守护。

动作：

- 补 parser、semantic analyzer、overview、capability graph、layout、UI controller、AI adapter 的测试。
- 增加大项目性能样本。
- 记录 10k+ 文件规模下的：
  - 扫描耗时
  - 解析耗时
  - 聚合耗时
  - 布局耗时
  - 峰值内存

验收标准：

- 每个核心模块都有自动化回归保护。
- 性能讨论不再停留在体感层面。

### 第 10 阶段，4 到 7 天：做增量分析和缓存

目标是基于已稳定的数据模型提升重复分析速度。

动作：

- 把流程拆成扫描、解析、聚合、layout 四层缓存。
- 缓存键至少包含：
  - 文件 hash
  - 编译命令
  - 项目配置版本
  - analyzer 版本
- 先做单项目重复分析加速，再补取消和失效策略。

验收标准：

- 第二次分析同一项目时不再全量重扫。
- 缓存命中和失效行为可解释、可验证。

### 第 11 阶段，5 到 8 天：做产品化输出能力

目标是让结果更接近可交付成果，而不只是调试产物。

动作：

- 补截图导出、分享包、审计报告模板。
- 增加多项目对比、版本前后差异能力。
- 输出建立在 L3 和证据链之上，而不是绕开它们。

验收标准：

- 输出物不只是 Markdown、DOT、JSON。
- 用户能产出可以直接拿去沟通的架构结果。

### 第 12 阶段，4 到 7 天：建立发布门槛和商业化配套

目标是让项目具备可发布、可维护、可合规的基础。

动作：

- 整理第三方依赖合规清单。
- 补安装、升级、恢复、日志采集、可观测性。
- 明确 AI 边界、隐私边界、离线/代理/自定义网关支持。
- 建立正式发布门槛。

验收标准：

- 每次发布前都必须通过：
  - 核心样本精度验证
  - 性能基线验证
  - UI smoke
  - 导出验证
  - 安装包验证

## 当前建议的直接下一步

不要继续把输出停留在“内部调试产物”，也不要在这个节点脱离产品结构去零散抠 UI。

推荐直接进入第 11 阶段：

1. 在现有 L2/L3、证据链和缓存之上补截图导出、分享包和审计报告模板。
2. 把多项目对比、版本前后差异和可交付输出作为下一轮主线。
3. 结合第 10 阶段已经稳定下来的数据链和性能模型，再开始系统化 UI 精修。

原因很简单：

- 第 10 阶段已经把重复分析性能和失效策略收口了。
- 现在适合把结果从“可分析”推进到“可交付、可沟通”。
- 再往后做产品化输出时，重复分析性能必须先稳定下来。
